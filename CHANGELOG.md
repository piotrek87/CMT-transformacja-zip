Zmiany w projekcie narzędzia migracyjnego Dataverse.

## 2026-03-13

- **Inicjalizacja repozytorium Git**: Projekt w wersji działającej – transformacja CMT z deduplikacją rekordów (naprawa błędu „Element o tym samym kluczu został już dodany” przy imporcie account/contact). Tag `v1.0-cmt-dziala` do powrotu do tego stanu. (piotrek87)

## 2026-03-12

- **CMT Transform – poprawne sprawdzanie pól w celu**: Przy usuwaniu pól nieistniejących w celu (StripFieldsNotInTarget) metadane encji z celu są teraz pobierane z `EntityFilters::Attributes`, dzięki czemu lista dozwolonych atrybutów jest wypełniana (wcześniej `Get-CrmEntityMetadata` bez tego filtra często zwracał `Attributes = null` i pola nie były usuwane – błąd „Missing Fields on Entity contact” przy imporcie). Dodane logowanie: pełny błąd połączenia z celem (Message + InnerException), dla każdej encji liczba dozwolonych atrybutów lub komunikat o braku atrybutów/błędzie metadanych. (Cursor)
- **Opcja 3 (Transformuj zip) w osobnym procesie**: Transformacja zipa (Transform-CMTZip) jest uruchamiana w osobnym procesie `powershell.exe`, tak jak opcja 2 (User Map) – moduł Microsoft.Xrm.Data.PowerShell nie ładuje się w hostach Cursor/VS Code (błąd „the module could not be loaded”). Skrypt transformacji przyjmuje opcjonalnie `-ConfigPath` i wtedy ładuje połączenie do celu z configu (bez przekazywania hasła w linii poleceń). (Cursor)
- **Opcja 3 – zip gotowy do importu (daty i ownerzy w zipie)**: Transformacja zipa (opcja 3) teraz: (1) w każdym rekordzie w data.xml ustawia **overriddencreatedon** na wartość **createdon** (oryginalna data utworzenia), a jeśli węzeł nie istnieje – dodaje go; (2) w data_schema.xml dodaje pole overriddencreatedon do encji, jeśli brakuje go w schemacie (żeby CMT importował datę); (3) ownerzy – jak dotąd – są podmieniani na podstawie IdMap (GUID źródło→cel). Zip po opcji 3 jest gotowy do importu bez konieczności wyboru User Map ani włączania Override created on w CMT. (Cursor)
- **Import CMT – właściciele i data utworzenia**: W README i w komunikacie po opcji 3 (Transform) dodane wyraźne przypomnienie: (1) przy imporcie w DataMigrationUtility trzeba wybrać plik **User Map** (Output\CMT_UserMap_ByDisplayName.xml), inaczej CMT ustawia wszystkich właścicieli na użytkownika wykonującego import; (2) żeby zachować oryginalne daty utworzenia, w CMT należy włączyć opcję Override created on. (Cursor)
- **CMT Transform – usuwanie pól także ze schematu i poprawki XML**: Przy StripFieldsNotInTarget: (1) usuwane są pola nieistniejące w celu także z pliku **data_schema.xml** w zipie (węzły `fields/field`), żeby weryfikacja schematu przy imporcie CMT nie zgłaszała „Missing Fields”; (2) przy usuwaniu z data.xml brana jest tylko nazwa z węzłów typu Element, z atrybutu `name` lub `Name` lub LocalName; (3) obsługa formatu CMT (elementy-pola jako dzieci `record`). (Cursor)

- **Przeprojektowanie aplikacji CMT**: Przepływ: (1) Buduj schemat importu – porównanie metadanych źródła i celu (Get-CommonEntitiesForCMT), wynik `data_schema_import.xml` z encjami i atrybutami tylko z celu. (2) Eksportuj dane ze źródła – CMT Export z tym schematem → zip. (3) Generuj User Map (imię+nazwisko) → User Map XML + `CMT_IdMap_SystemUser.json` (mapowanie GUID dla systemuser). (4) Transformuj zip – skrypt Transform-CMTZip.ps1 podmienia w zipie ID właścicieli według IdMap → `CMT_Export_ForTarget.zip`. (5) Użytkownik otwiera CMT (DataMigrationUtility), Import data, wybiera zip_ForTarget (i User Map). Menu: 1–Buduj schemat, 2–Eksport, 3–User Map, 4–Transformuj zip, 5–Pełna pipeline (1+2+3+4), 6–Instalacja modułu, 7–Schemat tylko ze źródła, 8–Test 1 rekord, 9–Import zip (skrypt). Nowe: Lib/Build-CMTImportSchema.ps1, Lib/Transform-CMTZip.ps1, Get-CommonEntitiesForCMT w Get-CMTSchemaFromSource.ps1, SystemEntitiesToSkip w CMTConfig. README zaktualizowany.

- **Opcja 7 (Generuj User Map) w menu CMT**: Uruchomienie w osobnym procesie PowerShell (powershell.exe), tak jak opcje 5 i 6 – moduł Microsoft.Xrm.Data.PowerShell wymaga standardowego hosta (błąd „PSRunspace-Host” w Cursor/VS Code).

- **CMT – migracja danych z datą, właścicielstwem po imieniu i nazwisku, stagami**: W README DataverseMigrationCMT dodana sekcja „Migracja danych przez CMT – co masz z tego, czego brakowało”: (1) data oryginalna tworzenia – schemat CMT z createdon/overriddencreatedon; (2) właścicielstwo po imieniu i nazwisku – nowy skrypt Lib\New-CMTUserMapByDisplayName.ps1 (łączy się ze źródłem i celem, dopasowuje systemuser po „Imię Nazwisko”, generuje plik User Map XML); (3) stagi na szansach – encje processstage, opportunitysalesprocess, leadtoopportunitysalesprocess w schemacie. W menu CMT dodana opcja 7: „Generuj User Map po imieniu i nazwisku”. Konfiguracja: UserMapFilePath w CMTConfig.ps1 lub -UserMapFilePath przy imporcie.

- **Domyślny tryb Upsert**: Migracja domyślnie w trybie Upsert (Create gdy brak rekordu na celu, Update gdy istnieje). MigrationConfig.ps1 i Start-DataverseMigration.ps1: MigrationMode = 'Upsert'; w Migrate-EntityData.ps1 fallback trybu to 'Upsert'.
- **Rozpoznawanie rekordu na celu po PK**: W trybie Upsert, gdy rekord nie jest w IdMap, wywoływane jest Get-TargetRecordIdByKey po kluczu głównym (GUID źródła). Jeśli w celu istnieje rekord z tym samym PK (np. po poprzedniej migracji z overriddencreatedon), wykonywany jest Update zamiast Create – eliminuje błędy „Cannot insert duplicate key”.
- **Filtrowanie atrybutów do schematu celu**: migratableAttrs po Get-MigratableAttributes jest filtrowane do atrybutów obecnych w TargetMeta. Atrybuty tylko na źródle (msdyn_*, msgdpr_*, msevtmgt_* itd.) nie trafiają do Create/Update – eliminuje błędy „entity doesn't contain attribute with Name = '...'”.
- **Convert-RecordToTargetAttributes – typy z celu**: Na początku funkcji budowane są mapy targetTypeMap i targetLookupTargets z TargetMeta. Dla każdego atrybutu używany jest typ z celu (gdy dostępny). Lookup/Customer/Owner: obsługa wartości ze źródła jako string (GUID) – parsowanie GUID i rozwiązywanie mapowania dla każdej dozwolonej encji docelowej (account, contact itd.). W fallbacku dla wartości string: konwersja według typu celu (Boolean→bool, OptionSet/State/Status→OptionSetValue, Money/Double/Decimal→decimal/Money, Lookup/Customer/Owner→EntityReference) – ogranicza „Incorrect attribute value type” i „Type Mismatch”.

- **Schemat tylko z celu**: Get-CommonEntities ustawia listę atrybutów do migracji wyłącznie na podstawie schematu celu (tgtAttrs), nie przecinania źródło–cel. Dzięki temu nigdy nie wysyłamy pól, których nie ma w celu (np. atrybuty z rozwiązań tylko na źródle).
- **TargetMeta zawsze w Copy-EntityRecords**: Start-DataverseMigration przekazuje TargetMeta do Copy-EntityRecords w każdym wywołaniu. Gdy TargetAttributeNames jest puste, targetAttrSet jest budowany z TargetMeta, więc lista pól do wysłania zawsze pochodzi ze schematu celu.
- **Typy atrybutów z metadanych celu**: Przed pętlą rekordów Copy-EntityRecords wypełnia z TargetMeta mapy entityAttrRequiresOptionSet, entityAttrRequiresBoolean i entityAttrRequiresLookup (OptionSet/Picklist/State/Status, Boolean/Bit, Lookup/Customer/Owner). Konwersja string→OptionSetValue, string→bool i string→EntityReference (z LookupIdMap) odbywa się przy budowaniu fieldsSet/fieldsCreate, co ogranicza błędy „Incorrect attribute value type” i „Type Mismatch” już przy pierwszej próbie.
- **activitypointer – PK activityid**: W Dataverse encja ActivityPointer ma klucz główny activityid, nie activitypointerid. We wszystkich miejscach (Copy-EntityRecords, Get-SourceRecordKeyValue, Get-EntityRecordCounts, Get-EntityRecordCounts w Get-EntityMetadata.ps1) dla activitypointer używany jest activityid. Przy budowaniu recHash z rekordu, jeśli brak klucza „id”, ustawiany jest z wartości atrybutu PK (np. activityid).
- **PK activityid dla wszystkich encji aktywności**: Encje email, task, appointment, phonecall, letter, fax, campaignresponse w Dataverse także używają activityid jako klucza głównego (nie emailid, taskid itd.). Dodana lista ActivityEntityNames i funkcja Get-PrimaryKeyAttributeName; Fetch i odczyt rekordu używają activityid dla tych encji (naprawa błędu „'Email' entity doesn't contain attribute with Name = 'emailid'”).
- **Pomijanie Create dla activitypointer**: API Dataverse nie obsługuje Create dla encji activitypointer (MessageProcessor.Empty). Copy-EntityRecords po pobraniu rekordów sprawdza EntityLogicalName -eq 'activitypointer' i zwraca od razu Created=0, Skipped=total, z komunikatem w logu, że aktywności migruje się przez email/task/appointment itd.

- Druga aplikacja migracyjna **DataverseMigrationCMT**: migracja przez Configuration Migration Tool (moduł Microsoft.Xrm.Tooling.ConfigurationMigration 1.0.0.88). Eksport ze źródła (Export-CrmDataFile) i import do celu (Import-CrmDataFile) z pliku schematu CMT; obsługa mapowania użytkowników (User Map) i opcji dat. Skrypty: Start-CMTMigration.ps1, Install-CMTModule.ps1, Config\CMTConfig.ps1, Lib\Connect-CMT.ps1, README (DataverseMigrationCMT).
- Obsługa błędów „Incorrect attribute value type System.String” (np. address2_addresstypecode, territorycode): zamiast rzucania wyjątku przy nieparsowalnej wartości – usunięcie problematycznego atrybutu z zestawu pól i ponowienie zapisu (rekord zapisuje się bez tego pola; dla kolejnych rekordów wartość string/int jest konwertowana na OptionSetValue). Konwersja int→OptionSetValue przy budowaniu fieldsSet/fieldsCreate dla atrybutów z entityAttrRequiresOptionSet. Regex na nazwę parametru rozszerzony o kropkę ([\w\.]+).
- Obsługa błędów „Type Mismatch … System.Boolean … System.String" (np. adx_identity_logonenabled): wykrywanie atrybutu z komunikatu (Attribute: Entity.attributename), konwersja string→bool (true/1/yes→true, false/0/no/''→false) i retry; przy nieparsowalnej wartości – usunięcie atrybutu z zestawu i ponowienie. Dla kolejnych rekordów konwersja string→bool przy budowaniu pól (entityAttrRequiresBoolean).
- Schemat celu jako podstawa listy pól: Get-CommonEntities gdy źródło ma pustą listę atrybutów używa listy atrybutów z celu (nie przecinania), więc przenoszone są tylko pola istniejące w schemacie celu. Copy-EntityRecords: fallback „z rekordu” działa tylko gdy jest lista atrybutów celu (TargetAttributeNames); dopuszczane są wyłącznie pola z tej listy. Gdy brak metadanych i brak listy celu – w logu informacja i przenoszone tylko overriddencreatedon / EntityMeta.
- Uzupełnianie pól przy braku metadanych: gdy migratableAttrs jest puste (źródło i cel bez atrybutów), Copy-EntityRecords pobiera pierwszy rekord ze źródła (Fetch z PK + createdon), wywołuje Get-CrmRecord z Fields „*” dla tego rekordu, odkrywa listę pól z odpowiedzi (z filtrem po TargetAttributeNames jeśli jest), po czym ponownie pobiera wszystkie rekordy z tymi polami – dzięki temu wartości (fullname, email itd.) trafiają do celu. Zawsze dodawany jest klucz główny (entityid) do listy pól Fetch.
- Liczniki postępu w CMD: Write-Progress w pętli rekordów (Migracja encji: contact, Rekord X z Y, utworzono/zaktualizowano/błędy) oraz na poziomie encji (Encja K z N: nazwa). W logu wpis postępu co 10 rekordów z liczbami utworzono/zaktualizowano/pominięto/błędy. Po zakończeniu pętli Write-Progress -Completed.
- Logowanie i liczenie bledow migracji: gdy Update lub Create konczy petle retry bez sukcesu – rekord liczony jako blad ($failed++) i w logu wpis „[REKORD N] Update/Create NIE POWIODL SIE po wszystkich probach – blad: …”. Dla pierwszych 3 rekordow lub pierwszej proby logowany tez kazdy blad API: „[REKORD N] Update/Create blad (proba K): …” – w logu od razu widać miejsce i tresc problemu.
- GUI: checkbox „Test: tylko 1 rekord na encje” przy przyciskach migracji – przy wlaczeniu migracja ograniczona do 1 rekordu na kazda encje (MaxRecordsPerEntity).
- Naprawa „Zakonczono: 0,0,0,0” przy 65 rekordach: wymuszenie tablicy przed pętlą (`$recordsToProcess = @($records)`), Get-SourceRecordsPaged zwraca `@($allRecords)`, log „Rozpoczynam petle migracji: N rekordow” oraz ostrzeżenie gdy lista rekordów jest pusta – eliminuje problem z enumeracją kolekcji zwracanej przez API.
